# 🚇 BakuCard Telemetry Pipeline

Welcome to the **BakuCard Telemetry Pipeline** project! This repository contains a backend data pipeline designed to process, clean, and analyze millions of daily trip records generated by validator devices across Baku's metro stations and bus networks.

---

## 📋 Scenario & Overview

Validator devices on the metro and buses push millions of trip records every day. However, real-world telemetric data is **dirty**—containing uncharged trips, corrupted negative fares, and blacklisted cards.

This project implements a high-performance data pipeline to:

1. **Clean** raw incoming records.


2. **Produce** real-time financial and passenger analytics.


3. **Decode** binary telemetry packets coming straight from physical devices.



### 🚫 Strict Performance Rules

To ensure optimal performance when handling millions of rows, this pipeline adheres to strict production constraints:

* **No Loops:** Classic `for`, `while`, `do-while`, and even `forEach` loops are strictly **banned**.


* **ES6 Structures Only:** Processing must rely entirely on functional methods like `.map()`, `.filter()`, and `.reduce()`.


* **Immutability:** The original raw data must never be mutated (no `.push()`, `.splice()`, or `.sort()` on inputs).


* **Variable Discipline:** No `var`—use `const` and `let` only.

## 🛠️ Pipeline Features

The pipeline in `pipeline.js` implements the following core tasks:

### 1. Data Cleaning (`cleanTrips`)

Filters out incomplete or corrupted trip data. A trip is considered "clean" only when `valid === true` and `fare > 0`.

### 2. Revenue Calculation (`totalRevenue`)

Accumulates the exact total revenue from all successfully cleaned trips using a highly efficient reduction pipeline.

### 3. Passenger Analytics & Blacklists (`uniquePassengers` & `isBlocked`)

* **`uniquePassengers`**: Isolates unique active passengers by mapping cards to an ES6 `Set` to eliminate duplicate records.


* **`isBlocked`**: Minimizes lookup times from $O(n)$ to $O(1)$ by utilizing hash-set lookups to instantly check if a passenger is on the blacklist.



### 4. Station Revenue Reports (`revenueByStation`)

Generates a `Map` that associates every physical station with its total calculated revenue.

### 5. Memory-Safe Device Caching (`createDeviceCache`)

Caches the last known status of validator hardware. Built using a `WeakMap` to prevent memory leaks—once a hardware device object is dropped by the system, its cache entry is automatically garbage-collected.

### 6. Double-Charging Prevention (`createProcessedRegistry`)

Uses a `WeakSet` registry to keep track of processed trip transactions. Ensures each trip object is submitted to the billing system exactly once, instantly rejecting duplicates.

### 7. Binary Telemetry Decoding (`decodeCounter` & `packCounter`)

* **`decodeCounter`**: Decodes a 10-byte binary packet representing passengers passing through a turnstile minute-by-minute. Outputs total passengers, the busiest minute, and active operational minutes.


* **`packCounter`**: Packs standard numbers into high-performance `Uint8Array` typed arrays.

## 💡 Technical Q&A: Integer Overflow in Typed Arrays

### **Q: What happens in `packCounter` if a number passed to the array is larger than 255?**

**A:** Because `Uint8Array` allocates exactly 8 bits (1 byte) per element, it can only store integers in the range of **0 to 255**.

If a number larger than 255 is passed (e.g., `256` or `257`), an **integer overflow** occurs. JavaScript handles this by wrapping the value back around using modulo-256 arithmetic ($value \pmod{256}$):

* `256` wraps around and is stored as `0`
* `257` wraps around and is stored as `1`
* `512` wraps around and is stored as `0`
